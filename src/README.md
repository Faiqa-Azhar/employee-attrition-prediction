## 1. Project Overview

### The Problem
Employee attrition costs organisations **50–200% of an employee's annual salary**
in recruitment, onboarding, training, and lost productivity. Most HR teams only
react after a resignation letter arrives — by which point intervention is too late.

### The Solution
An end-to-end binary classification pipeline that predicts whether an individual
employee **Will Leave** or **Will Stay**, assigns a **Flight Risk Probability**,
explains *why* using live SHAP feature attribution, and calculates the **ROI
of a retention intervention** — giving HR teams the information to act early
and act precisely.

### Target Users
| User | Value Delivered |
|---|---|
| HR Business Partners | Know which employees to approach, and why, before they resign |
| People Analytics Teams | Reproducible, explainable attrition model with documented methodology |
| ML Engineers | Reference implementation of leakage-free imbalanced classification pipeline |

### Measured Impact
| Metric | Value |
|---|---|
| At-risk employees caught by model | **85%** (Recall = 0.85) |
| Class imbalance handled | 84% Stay / 16% Leave → SMOTE inside training folds |
| Replacement cost avoided per retained employee | **USD 30,000 – USD 120,000** |
| ROI on a USD 3,000 retention intervention | **3,900%** |
| Models compared | 3 (Logistic Regression, Random Forest, XGBoost) |

## 3. Key Features

- **Leakage-Free SMOTE Pipeline** — Synthetic oversampling is applied strictly
  inside each cross-validation training fold using `imblearn.ImbPipeline`.
  The test set is never touched by SMOTE — eliminating the most common
  methodology error in imbalanced classification. Most comparable projects
  on Kaggle apply SMOTE before splitting, contaminating test metrics.

- **Recall-First Model Selection with Threshold Tuning** — All models are
  tuned and selected using Recall as the primary metric, not accuracy.
  A threshold sweep from 0.10 to 0.50 enforces Recall ≥ 0.70 with F1 as
  a tiebreaker — reflecting the real business cost asymmetry where missing
  a leaver is 50–200× more expensive than a false alarm.

- **7 Engineered Interaction Features** — Beyond the 35 raw IBM HR features,
  seven domain-driven interaction features are created: Promotion Stagnation
  Index, Income-per-Experience ratio, Career Velocity, Loyalty Score,
  Satisfaction Composite score, Overtime Burden Flag, and Role Stability ratio.
  No comparable project on this dataset creates interaction features.

- **Live Per-Employee SHAP Explainability** — Every prediction generates a
  real-time SHAP bar chart computed against a background reference sample
  from training data, showing exactly which factors pushed this specific
  employee's risk score up (red) or down (blue).

- **SHAP-Driven HR Recommendations** — Recommendations are generated by
  ranking that employee's live SHAP values and mapping the top risk drivers
  to specific, actionable HR interventions. Protected characteristics
  (marital status) are explicitly excluded from recommendations despite
  being statistically significant predictors.

- **Retention ROI Calculator** — The Streamlit app calculates conservative
  (50% salary) and realistic (200% salary) replacement cost scenarios,
  net savings, and percentage ROI for any user-specified intervention cost —
  giving HR managers a quantified business case in seconds.

### Data Flow

```
Raw CSV
  → FeatureEngineer.transform()   drop constants, map binary/ordinal,
                                   create 7 interaction features
  → ColumnTransformer              OneHotEncode nominal columns
  → SMOTE                          training folds ONLY — never test set
  → Model.fit()                    RF / XGB / LogReg with CV tuning
  → predict_proba()                continuous probability score
  → Threshold (0.30)               tuned for Recall ≥ 0.70
  → SHAP Explainer                 per-employee attribution vs background
  → Streamlit UI                   prediction + recommendations + ROI
```

### Key Design Decision — Why ImbPipeline
Standard `sklearn.Pipeline` does not support resamplers. `imblearn.ImbPipeline`
ensures SMOTE runs only during `.fit()` inside each CV fold — never during
`.predict()` or `.transform()` on the test set. This is the correct
implementation; most published notebooks apply SMOTE before splitting,
which artificially inflates all reported metrics.

## 5. Getting Started (Local Development)

### Prerequisites
```
Python     3.10 or higher
pip        23.0 or higher
Git        2.30 or higher
```

### Installation

```bash
# 1. Clone the repository
git clone https://github.com/Faiqa-Azhar/employee-attrition-prediction.git
cd employee-attrition-prediction

# 2. Create and activate virtual environment
python -m venv venv

# Windows
venv\Scripts\activate

# Mac / Linux
source venv/bin/activate

# 3. Install all dependencies
pip install -r requirements.txt
```

### Dataset Setup
Download the IBM HR Analytics dataset from Kaggle:
https://www.kaggle.com/datasets/pavansubhasht/ibm-hr-analytics-attrition-dataset

Place the file at:
```
data/raw/HR_Attrition.csv
```
### Run EDA Notebook

```bash
# From project root
jupyter notebook notebooks/01_eda.ipynb
```

Select the project `venv` as the Jupyter kernel. Run All Cells.
Saves figures to `reports/figures/`.


### Train All Models

```bash
cd src
python train.py
```

```bash

# Predict on a CSV of employees

                  --output ../data/processed/predictions.csv \
                  --threshold 0.30
```

### Environment Configuration
No `.env` file or external secrets are required. All path configuration
lives in `src/utils.py` and is derived automatically from the project root:

```python
# src/utils.py
BASE_DIR      = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
RAW_DATA_PATH = os.path.join(BASE_DIR, "data", "raw", "HR_Attrition.csv")
MODELS_DIR    = os.path.join(BASE_DIR, "models")
RANDOM_STATE  = 42
```

- All new transformers must inherit from `BaseEstimator, TransformerMixin`
- All new features must include a business rationale comment
- `RANDOM_STATE = 42` must be used for all stochastic operations
- Test set must never be touched until final evaluation

> Accuracy is not reported as a headline metric. On an 84/16 imbalanced dataset,
